# Coding Style

- Comments only on complex code -- if the logic is self-evident, skip the comment
- Prefer editing existing files over creating new ones
- Minimal impact -- changes should only touch what is necessary
- Find root causes, not temporary fixes
- Do not add docstrings, type annotations, or comments to code you did not change
- Do not add error handling for scenarios that cannot happen
- Three similar lines of code is better than a premature abstraction
- Do not over-engineer: no feature flags, no backwards-compatibility shims, no hypothetical future requirements
